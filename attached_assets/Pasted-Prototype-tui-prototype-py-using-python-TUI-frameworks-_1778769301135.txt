The table is always visible above a footer/input line. The table has exactly one editable column (the proposed value). The footer operates in two modes with different arrow-key semantics, because focus lives in different places in each:

- **Row-selection mode** — Focus is in the footer line, with the cursor appearing after a colon. Up/down arrows scroll the table viewport a full page, allowing the user to scan rows that extend beyond the viewport before deciding what to edit. Typing a row number and pressing Enter fast-jumps to that row and activates cell-edit mode. `a` accepts all when no unresolved rows remain. PageUp/PageDown also scroll the viewport.
- **Cell-edit mode** — Focus shifts to the selected row's editable cell in the CURRENCY column. Up/down arrows move the table cursor row-by-row. The footer shows `Enter to confirm · Esc to cancel`. Validation runs on every keypress: a red error replaces the instruction line and Enter is blocked until the value is valid. Esc returns to row-selection mode without committing. PageUp/PageDown scroll the viewport in this mode too. The table updates in place after each confirmed edit, with collision and validation status re-evaluated reactively. Footer shows position and remaining work, e.g.: `Row 47 of 252 · 18 unresolved`.

What to try:
  - The table opens with two COLLISION rows in red                                                         
  - Press ↑/↓ — the viewport scrolls by page (four rows won't scroll, but you'll see it work on the fixture if more rows are added)
  - Type 1 then Enter — jumps to row 1, Input fills with AT-T, hint changes to Enter confirm · Esc cancel · ↑↓ move row
  - Type an invalid symbol (e.g. 1BAD) — red validation error appears inline, Enter is blocked
  - Type ATT — validation clears, Enter works, table updates with green ✓
  - Repeat for row 2 — once both collisions are resolved, a accepts and prints the summary
  - Esc from row-selection mode exits with Cancelled
