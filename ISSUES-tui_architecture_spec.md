Issues with tui_architecture_spec.md
High Priority
1. The spec does not define the full state machine. It lists modes at line 9 (line 9), but it does not model distinct confirmation intents: “skip adding commodities” after ctrl+c with collisions, versus “accept all” after collisions reach zero. This will cause implementers to collapse two different flows from storyboard frames 1b and 6/14.
1. acceptAllConfirm ownership is contradictory. Root mutates it at lines 58 and 71 (line 58), Prompt owns it as local state at line 90 (line 90), and Footer receives it as a prop at line 181 (line 181). Make one owner explicit, likely root/app state.
1. ctrl+c behavior is almost absent from the dispatcher. The storyboard requires first ctrl+c to enter exit confirmation when collisions remain, second ctrl+c to send SIGINT, and normal cancel behavior elsewhere. The spec only renders header text and does not define event handling.
1. Filtering is underspecified. Lines 55 and 133 (line 55) mention Tab, !, and “non-metafilter portion,” but do not define the query grammar: Tab as collision metafilter, ! equivalent, matching columns, case rules, highlight spans, empty-result selection, or how ctrl+s behaves while a filter is active.
1. Edit input behavior contradicts the storyboard. Line 161 (line 161) says typed characters append, but the storyboard requires streaming overwrite against ghost text, with deviation detection. This needs an explicit edit-buffer algorithm.
1. Medium Priority
1. 6. The domain model is too lossy. isCollision: boolean at line 23 (line 23) cannot express collision groups, default safe values, current edited value versus original target, source-derived safe values, or why collision icons disappear live while editing.
1. 
Sorting rules from the storyboard are missing. Implementers need the stable ordering rule: sort by sanitized GnuCash source, then ASCII original source, and do not reorder when Beancount Token values change.
1. Layout needs a deterministic render contract. Line 130 (line 130) gives rough columns, but the spec needs exact terminal-line allocation for 15x75, Unicode width rules, clearing behavior for inline rendering, filler blank lines, and no-wrap assumptions.
1. Validation is incomplete. The spec references VALID/INVALID, but does not define Beancount token grammar, when ✓ appears, when ✗ appears, submit gating, or the 24th/25th character behavior from storyboard frame 11.
1. Source-selection behavior is not exact enough. The storyboard requires ↑/↓ wrapping through source rows, live pointer movement when input exactly matches a source safe value, tab autocomplete, and retention of the last explicit source pointer when there is no match. The spec only partially covers this at lines 142 and 169 (line 142).
1. Recommended Fix
1. Rewrite the document as an implementation contract, not component prose. The next version should have these sections: state machine table, event matrix, derived selectors, render grid, filtering grammar, edit-buffer algorithm, validation rules, scrolling rules, collision algorithm, and golden tests mapped frame-by-frame to the storyboard.
